during record save. If a workflow performs a field update, it causes the record to be updated again,

which re-executes before and after update triggers within the same transaction. This is a common cause

Why Workflow Causes Trigger to Run Again
  • Because workflow field update performs internal update.
  • Salesforce treats it like a new update event.
Thus:
  • Trigger.isUpdate = true
  • Trigger.oldMap contains pre-workflow values
Example Problem
  • Trigger

List<Contact> contacts = new List<Contact>();
for(Account acc : Trigger.new){
contacts.add(new Contact(

));
}
insert contacts;
}

Workflow Only Fires Once
  • Even if workflow field update happens, workflow itself does not run repeatedly in loop.
  • But trigger runs again.
Governor Limits Shared
  • Workflow + Trigger share:
  • CPU time
  • SOQL limit
  • DML limit
  • Heavy workflow logic increases CPU usage.
Rollback Behavior
  • If after trigger throws exception:
  • Workflow actions rolled back
  • Entire transaction rolled back
  • Because commit not happened yet.
Workflow vs Flow
  • Modern Salesforce:
  • Workflow being phased out
  • Record-triggered Flow preferred
  • But many legacy orgs still use workflow.
  • Must understand both.
Enterprise Best Practice
Avoid:
  • Workflow field updates on same object handled by trigger
Prefer:
  • Single automation layer per object
  • Consolidate into Flow or Trigger
Debugging Workflow Interaction
  • If trigger fires twice unexpectedly:
  • 1Check Workflow Rules.
  • 2Check field updates in workflow.
  • 3Check debug logs for second execution.
  • 4Add debug counter variable.
  • 5Use field-change detection.
When NOT to Use Workflow with Trigger
  • Avoid workflow when:
  • Complex logic needed
  • Multiple field updates
  • Heavy integration logic
  • Large data volume
  • Prefer Flow or Apex.
