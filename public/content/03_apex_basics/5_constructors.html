30-Second Interview Answer

A constructor in Apex is a special method used to initialize a class when an object is created. It has the

same name as the class and does not have a return type. Constructors are used to set initial values,

  • initialize variables, or perform setup logic when an instance is instantiated.
2-Minute Technical Explanation
  • In Apex:
  • Constructor name must match class name.
  • It runs automatically when new keyword is used.
  • It does not return any value.
A class can have multiple constructors (constructor overloading).
  • If no constructor is defined:
‚Üí Salesforce provides a default no-argument constructor.
Real-Time Scenario
  • Suppose you're building a service class for processing Orders.
  • When object created:
  • You want to initialize configuration.
  • Load custom metadata.
  • Set default values.
  • Use constructor to prepare class before execution.

Basic Constructor Example
public class Employee {
public String name;
public Employee(String empName) {
name = empName;
}
}
Usage:
Employee emp = new Employee('Rushikesh');
System.debug(emp.name);
Default Constructor
  • If you define:
public class MyClass {
}
  • Salesforce internally creates:
public MyClass() {}
Constructor Overloading
public class AccountService {
public String region;
public AccountService() {
region = 'India';
}
public AccountService(String regionName) {
region = regionName;
}
}
Usage:
AccountService a1 = new AccountService();
AccountService a2 = new AccountService('US');

public class SingletonService {
private static SingletonService instance;
private SingletonService() {}
public static SingletonService getInstance(){
if(instance == null){
instance = new SingletonService();
}
return instance;
}
}
  • Ensures only one instance per transaction.
üìå Constructor in Controller (LWC/Aura/VF)

System.debug('Controller initialized');

}
  • Constructor runs when controller loads.
Constructor Execution Flow
When:
MyClass obj = new MyClass();
Execution:
  • 1Memory allocated.
  • 2Constructor runs.
  • 3Variables initialized.
  • 4Object returned.
Performance Considerations
Avoid:
  • SOQL in constructor.
  • DML in constructor.
  • Heavy loops in constructor.
Because:
  • Constructor runs every time object instantiated.
Better:
  • Use lazy loading method.
üìå Lazy Loading Pattern (Better Practice)
public class ConfigService {
private Map<String, String> configMap;
public Map<String, String> getConfig(){
if(configMap == null){
configMap = loadConfig();
}
return configMap;
}
}
  • Loads only when needed.
Constructor Not Called in Static Method
  • If class only contains static methods:
  • No object created ‚Üí constructor never runs.
Constructor in Test Class
  • Test class also supports constructor, but rarely needed.
Recursive Instantiation
  • Avoid creating new instance inside constructor repeatedly.
Debugging Constructor Issues
  • If logic not running:
  • Check if object actually instantiated.
  • Check access modifier.
  • Check if static method used instead.
  • Check test execution context.
When NOT to Use Constructor
  • Avoid using constructor for:
  • Integration callouts
  • DML
  • Expensive SOQL
  • Heavy logic
Instead:
  • Call dedicated initialization method.
