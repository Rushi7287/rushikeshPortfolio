üîµ 1.c Governor Limits

üé§INTERVIEW SECTION
30-Second Interview Answer

Governor Limits are runtime limits enforced by Salesforce to ensure fair resource usage in its

multitenant architecture. They restrict things like SOQL queries, DML operations, CPU time, heap size,

and callouts per transaction. If a limit is exceeded, Salesforce throws a runtime exception and the

  • transaction fails.
2-Minute Technical Explanation

Because Salesforce is multitenant, all orgs share the same infrastructure. To prevent one org from

consuming excessive CPU, memory, or database resources, Salesforce enforces limits per transaction.

  • Governor limits apply mainly to:
SOQL queries (100 sync / 200 async)
DML operations (150 per transaction)
Records processed (10,000 in DML)
CPU time (10 sec sync / 60 sec async)
Heap size (6 MB sync / 12 MB async)
Callouts (100 per transaction)
If any limit exceeds ‚Üí System.LimitException (uncatchable).
Real-Time Scenario
  • You write:
for(Account acc : Trigger.new){
Account a = [SELECT Id FROM Account WHERE Id = :acc.Id];
}
  • If 200 records inserted ‚Üí 200 SOQL queries.
  • Limit is 100.
Result:
  • System.LimitException: Too many SOQL queries: 101
  • Entire transaction rolls back.
Tricky Interview Questions
  • 1Can we catch a LimitException?
No. It is uncatchable.
  • 2Why async has higher limits?
  • Because it runs separately and does not block user request.
  • 3Do limits reset inside trigger?
No. They are per transaction.
  • 4Does calling another class reset limits?
No.
  • 5Does Batch Apex reset limits?

  • Use Batch for >50k records.
  • Use Queueable for chaining.
  • Use selective queries.

  • Complex report processing

üîµ 1.d Environments (Dev, QA, UAT, Prod)
üé§INTERVIEW SECTION
30-Second Interview Answer

Salesforce environments represent different stages of the application lifecycle: Development, QA, UAT,

and Production. Development is used for building features, QA for testing functionality, UAT for

business validation, and Production is the live environment used by end users. Salesforce provides

  • sandboxes to replicate production safely for development and testing.
2-Minute Technical Explanation
Salesforce follows a structured SDLC (Software Development Life Cycle).
üîπ Development (Dev)
  • Where developers build features.
  • Usually Developer Sandbox.
  • Unit testing happens here.
  • No real production users.
üîπ QA (Quality Assurance)
  • Full functional testing.
  • Integration testing.
  • Regression testing.
  • Usually Partial or Full Sandbox.
üîπ UAT (User Acceptance Testing)
  • Business users validate requirements.
  • Test real-life scenarios.
Usually Full Sandbox (with real data).
Production
  • Live environment.
  • Real users.
  • Real data.
  • Strict governance.
üî• Real-Time Project Flow (Example)
  • Let‚Äôs say you build:
  • "Automatic discount logic on Opportunity"
  • 1Developer builds in Dev sandbox.
  • 2Deploy to QA ‚Üí testers validate logic.
  • 3Deploy to UAT ‚Üí business confirms expected behavior.
  • 4Deploy to Production ‚Üí feature goes live.
