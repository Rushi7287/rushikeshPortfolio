30-Second Interview Answer

A Wrapper Class in Apex is a custom class used to combine multiple data types or objects into a single

structure. It is commonly used for UI communication (LWC/Aura), complex JSON handling, and grouping
  • related data together for processing.
2-Minute Technical Explanation
  • Wrapper classes are used when:
  • You need to combine multiple sObjects
  • You need to send structured data to LWC
  • You need to represent nested JSON
  • You want to add extra fields not present in database
  • You want to group data logically
  • Example use case:
  • Send Account with related Contacts and total count to LWC.
  • Instead of sending separate lists ‚Üí
  • Create wrapper class.
Real-Time Scenario
Requirement:
  • Display Accounts in LWC with:
  • Account Name
  • Number of Contacts
  • Total Opportunity Amount
  • This structure does not exist in DB directly.
  • Create wrapper class.

Yes.

‚Üí Yes (with @AuraEnabled).
  • 3Is wrapper stored in database?

‚Üí Wrapper is runtime structure.
  • 5Can wrapper be used in JSON parsing?
‚Üí Yes.
Senior-Level Discussion Points
  • UI-optimized DTO pattern
  • Wrapper vs Map usage
  • JSON wrapper for integration
  • Performance consideration
  • Immutable wrapper pattern
  • Clean architecture separation

9.a Custom Metadata Types (CMDT)
üé§INTERVIEW SECTION
30-Second Interview Answer
Custom Metadata Types (CMDT) allow developers to store configuration data as metadata rather than
  • data. Unlike Custom Settings, CMDT records are deployable between environments and can be

packaged, making them ideal for storing business rules, feature flags, and integration configurations.

2-Minute Technical Explanation
  • Custom Metadata Types are used when:
  • You need configurable logic
  • Admins should control behavior
  • Configuration must move across environments
  • Avoid hardcoding values in Apex
  • Example use cases:
  • API endpoints
  • Discount percentages
  • Feature toggles
  • Validation rules configuration
  • Integration mapping
  • Query CMDT like normal object:
  • SELECT DeveloperName, Endpoint__c
  • FROM Integration_Config__mdt
  • Key advantage:
  • Metadata ‚Üí deployable via change sets or packages.
Real-Time Scenario
Requirement:
  • Different ERP endpoints for:
  • Dev
  • QA
  • Prod
  • Instead of hardcoding URL in Apex:
  • Store in CMDT.
  • Apex dynamically reads endpoint based on environment.
  • No code change needed.
Tricky Interview Questions
  • 1Can CMDT records be deployed?
Yes.
  • 2Can we modify CMDT via Apex?
‚Üí ‚ùå Not directly in runtime (only via Metadata API).
  • 3Difference between Custom Setting and CMDT?
‚Üí CMDT deployable, Custom Setting not.
  • 4Does CMDT count toward SOQL limits?
‚Üí Yes.
  • 5Can CMDT be packaged?
‚Üí Yes.
Senior-Level Discussion Points

  • Feature flag implementation
  • Dynamic validation engine

üî∑ 9.b Custom Settings (Hierarchy / List)
üé§INTERVIEW SECTION
30-Second Interview Answer

Custom Settings are a legacy configuration mechanism in Salesforce used to store application settings.

They come in two types: List Custom Settings and Hierarchy Custom Settings. Hierarchy settings

allow user/profile/org-level overrides, while List settings store static configuration records. Today,

  • Custom Metadata Types are preferred over Custom Settings.
2-Minute Technical Explanation
  • Custom Settings are used for:
  • Storing configuration data
  • Avoiding hardcoded values
  • Controlling logic dynamically
  • Two types:
List Custom Settings
  • Store multiple records
  • Similar to custom object
  • No hierarchy override
Hierarchy Custom Settings
  • Allow override at:
  • Org level
  • Profile level
  • User level
  • Hierarchy resolution order:
  • User ‚Üí Profile ‚Üí Org default
  • Major difference from CMDT:
Real-Time Scenario
Requirement:
  • Disable trigger logic for specific user during data migration.
  • Use Hierarchy Custom Setting:
Field: Disable_Trigger__c
  • For migration user:
  • Set value = true
  • In trigger:
if(MySetting__c.getInstance().Disable_Trigger__c){
return;
}
  • Now only that user bypasses logic.
