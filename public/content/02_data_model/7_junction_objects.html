2.g Junction Objects

üé§INTERVIEW SECTION
30-Second Interview Answer

A Junction Object is a custom object used to create a many-to-many relationship between two objects. It

contains two Master-Detail relationships, each linking to one of the parent objects. This allows a record

  • from one object to relate to multiple records from another object.
2-Minute Technical Explanation
  • Salesforce does not directly support many-to-many relationships.
  • To implement many-to-many:
We create a third object (Junction Object).
Structure:
  • Parent A ‚Üê Master-Detail ‚Üí Junction Object ‚Üê Master-Detail ‚Üí Parent B
Example:
  • Student__c
  • Course__c
  • One student can enroll in multiple courses.
  • One course can have multiple students.
Solution:
Create Enrollment__c (Junction Object)
Fields:
Student__c (Master-Detail)
Course__c (Master-Detail)
Now:
  • Student ‚Üî Course many-to-many achieved.
Real-Time Scenario
  • Project Assignment System:
  • Employee__c
  • Project__c
  • One employee can work on multiple projects.
  • One project can have multiple employees.
Create:
  • Project_Assignment__c
Add:
Employee__c (Master-Detail)
Project__c (Master-Detail)
  • Add fields:
  • Role__c
  • Start_Date__c
  • Hours__c
  • Now flexible and scalable.
Tricky Interview Questions
  • 1Can junction object have Lookup instead of Master-Detail?
Yes, but then not true many-to-many with roll-ups.
  • 2Can roll-up summary exist?
‚úÖ Yes (on both parents).
  • 3How many Master-Detail allowed?
  • Maximum 2 per object.
  • 4What happens if one parent deleted?
  • Junction record deleted.
  • 5Who owns junction record?
  • It inherits ownership from primary master.
Senior-Level Discussion Points
  • Primary Master concept
  • Sharing inheritance
  • Data skew issues
  • Roll-up performance
  • Reparenting design
  • Large scale enrollment models

  • Custom object with two Master-Detail fields.

  • In junction object:
  • First created Master-Detail becomes:
  • Primary Master

Roll-Up Summary Capability
  • Because both are Master-Detail:
  • You can create:

  • If Student deleted:
‚Üí All Enrollment records deleted.
  • If Course deleted:
‚Üí All Enrollment records deleted.
Performance Considerations
  • ‚ö† Data Skew Risk:
If:
  • One Course has 100,000 students
Then:
  • Lock contention
  • Sharing recalculation delay
  • Roll-up performance issues
  • Best practice:
  • Avoid massive skew under single parent.
Changing Master Order
  • Cannot change primary master easily after creation.
Converting to Lookup
  • Roll-ups lost.
Sharing Behavior
  • Junction inherits sharing from primary master.
Debugging Junction Issues
  • If relationship not visible:
  • 1Check both Master-Detail fields created.
  • 2Check child relationship name.
  • 3Check sharing inherited.
  • 4Check reparenting setting.
  • 5Check roll-up calculation delay.
üìå Lookup-Based Many-to-Many (Alternative)
Sometimes:
  • Use two Lookups instead of Master-Detail.
  • Used when:
  • Independent ownership required.
  • No roll-up needed.
  • Flexible sharing required.
  • But not strict many-to-many.
Best Practices
  • Use Master-Detail for true many-to-many.
  • Carefully choose primary master.
  • Avoid parent data skew.
  • Name child relationship properly.
Add additional attributes on junction (role, date, etc.).
When NOT to Use Junction Object
  • If relationship:
  • Simple one-to-many ‚Üí Use Master-Detail
  • Optional ‚Üí Use Lookup
  • No additional fields required ‚Üí Maybe avoid extra object

üé§INTERVIEW SECTION
30-Second Interview Answer

Apex is a strongly typed, object-oriented programming language provided by Salesforce to execute

business logic on the Lightning Platform. It is similar to Java and runs on Salesforce servers. Apex is used

to write triggers, classes, batch jobs, integrations, and custom logic while respecting governor limits.

2-Minute Technical Explanation
  • Apex is:
  • Server-side language
  • Strongly typed
  • Case insensitive
  • Runs in multitenant environment
  • Subject to governor limits
  • It supports:
  • Classes
  • Methods
  • Variables
  • Collections
  • DML
  • SOQL
  • Exception handling
  • Asynchronous processing
  • Apex executes in:
  • Triggers
  • Anonymous window
  • REST endpoints
  • Batch/Queueable jobs
Controllers (LWC/Aura/VF)
Real-Time Scenario
Requirement:
  • When Opportunity is Closed Won:
‚Üí Create Project record automatically.
Solution:
  • Write Apex Trigger or Apex Class.
Because:
  • Declarative automation may not handle complex logic.
Tricky Interview Questions
  • 1Is Apex compiled or interpreted?
  • Compiled and executed on Salesforce runtime engine.
  • 2Does Apex run client-side?
‚ùå No (server-side).
  • 3Can Apex bypass governor limits?
No.
  • 4Is Apex synchronous by default?
Yes.
  • 5Can Apex directly access database?
Senior-Level Discussion Points
  • Execution context
  • Transaction boundaries
  • Heap size
  • With sharing vs without sharing
  • Synchronous vs asynchronous CPU usage
  • Platform event handling
