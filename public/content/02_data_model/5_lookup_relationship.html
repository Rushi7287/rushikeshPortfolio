2.e Lookup Relationship

üé§INTERVIEW SECTION
30-Second Interview Answer

A Lookup Relationship in Salesforce is a loosely coupled relationship between two objects where one

object references another via a lookup field. The child record can exist independently of the parent, and

  • deleting the parent does not automatically delete the child.
2-Minute Technical Explanation
  • Lookup is used when:
  • Relationship is optional.
  • Objects are loosely connected.
  • Child should survive even if parent is deleted.
Example:
Contact ‚Üí Account (Lookup)
Opportunity ‚Üí Campaign (Lookup)
  • Custom Object ‚Üí Account
  • Key characteristics:
Parent deletion does NOT delete child (unless configured otherwise).
  • Child can exist without parent.
  • No Roll-Up Summary fields supported.
  • Sharing is independent.
Real-Time Scenario
Suppose:
  • You create custom object:
  • Site_Visit__c
  • It has lookup to:
  • Engineer__c
  • If Engineer is deleted:
Site Visit record still exists (Engineer field becomes null).
  • Business rule:
  • Site Visit must not be deleted automatically.
  • So use Lookup.

What Is Lookup Internally?
  • Lookup field stores:
  • Parent record Id
  • Acts like foreign key
  • Indexed field
  • Database-level relationship but loosely enforced.
Example Structure
  • Child Object:
  • Site_Visit__c
Field:
Engineer__c (Lookup to Engineer__c)
Internally:
  • Site_Visit__c
  • Id
  • Name
Engineer__c (stores Engineer Id)
Relationship Query Example
  • Parent to Child
SELECT Name, (SELECT Name FROM Site_Visits__r)
  • FROM Engineer__c
  • Child to Parent
  • SELECT Name, Engineer__r.Name
  • FROM Site_Visit__c
Deletion Behavior
  • If parent deleted:
  • Child record:
  • Remains
  • Lookup field becomes null
  • Unless ‚ÄúDelete this record also‚Äù option enabled
Sharing Behavior
Lookup:
  • Child does NOT inherit parent sharing.
  • Child has its own OWD.
  • Can have separate sharing rules.
Lookup Filter
  • Used to restrict which records can be selected.
Example:
  • Only show Engineers where Status = Active.
Helps:
  • Data integrity
  • Better UI control
Performance Considerations
  • Lookup field is indexed.
  • Filtering on lookup field is selective.
  • Good for LDV.
  • Bad practice:
  • Cross-object formula referencing deep hierarchy.
Converting Lookup ‚Üí Master-Detail
Conditions:
  • No existing orphan records
  • Child OWD must be Public Read/Write
  • No roll-up summary dependent on it
Deleting Parent
  • Child may become orphan record.
Large Data Volume
  • Relationship queries may hit heap limit.
Debugging Lookup Issues
  • If relationship not working:
  • 1Check lookup filter.
  • 2Check field-level security.
  • 3Check required setting.
  • 4Check sharing.
  • 5Check cascade delete option.
Lookup vs Master-Detail Quick Comparison
  • Feature Lookup Master-Detail
  • Parent Required Optional Required
Sharing Inherited ‚ùå ‚úÖ
  • Cascade Delete Optional Always
When NOT to Use Lookup
If:
  • Child must always depend on parent
  • Roll-up summary required
  • Strict data integrity needed
  • Then use Master-Detail.
