2.c Fields

üé§INTERVIEW SECTION
30-Second Interview Answer

Fields in Salesforce are columns in an object that store specific data values. They define the type of

information stored in records, such as Text, Number, Date, Picklist, Lookup, or Formula. Fields can be

standard or custom, and they control data structure, validation, security, and reporting behavior.

2-Minute Technical Explanation
  • Fields define what type of data is stored inside an object.
  • There are two main types:
Standard Fields
  • Provided by Salesforce
Example: Name, CreatedDate, OwnerId
  • Cannot be deleted
Custom Fields
  • Created by user
  • API name ends with __c
  • Can be deleted
  • Fully customizable
  • Field Types Examples
  • Text
  • Long Text Area
  • Number
  • Currency
  • Date / DateTime
  • Checkbox
  • Picklist
  • Multi-Select Picklist
  • Lookup
  • Master-Detail
  • Formula
  • Roll-Up Summary
  • Email / Phone / URL
  • Auto Number
  • External ID
Real-Time Scenario
  • You create custom object:
  • Project__c
  • Add fields:
Budget__c (Currency)
Start_Date__c (Date)
Status__c (Picklist)
Client__c (Lookup to Account)
  • Now each record stores structured business data.

Field Architecture Internally
  • Each field has metadata:
  • Data type
  • Length
  • Required flag
  • Default value
  • FLS settings
  • Index flag
  • Relationship reference
  • Stored in metadata engine.
Field Categories
  • Ûæ†Æ Data Fields
  • Store actual values.
Example:
  • Name, Amount, Industry
  • Ûæ†Ø Relationship Fields
  • Connect objects.
  • Lookup
  • Master-Detail
  • Ûæ†∞ Calculated Fields
  • Formula
  • Roll-Up Summary
Text Field
  • Stores string values.
  • Max length: 255 characters.
Picklist
  • Restricted list of values.
  • Better than free text.
Multi-Select Picklist
  • Stores multiple values.
  • ‚ö† Hard to query.
Formula Field
Example:
  • Amount * 0.18
  • Calculated at runtime.
  • No storage used.
Roll-Up Summary
  • Available only in Master-Detail.
Example:
  • Sum of child Opportunity Amounts on Account.
Lookup Field
  • Loose relationship.
  • Child can exist without parent.
Master-Detail
  • Strong relationship.
  • Child deleted if parent deleted.
  • Supports Roll-Up Summary.
External ID
  • Used for:
  • Integration
  • Upsert operation
  • Unique identification
Indexing
  • Indexed fields:
  • Id
  • Name
  • External ID
  • Unique fields
  • Primary lookup fields
  • Custom index can be requested.
Performance Considerations
  • Bad Practices:
  • Too many formula fields
  • Non-selective filters
  • Multi-select picklist in query filter
  • Good Practices:
  • Use External ID for integration
  • Use indexed fields in WHERE clause
  • Avoid heavy cross-object formula
Changing Field Type
  • May cause data truncation.
Deleting Field
  • Data goes to recycle bin for 15 days.
Required Field + Integration
  • Integration may fail if required field missing.
Debugging Field Issues
  • If record fails:
  • 1Check required fields
  • 2Check validation rules
  • 3Check FLS
  • 4Check field type mismatch
  • 5Check formula errors
Storage Impact
  • Text fields consume storage.
  • Long text consumes more.
  • Attachments consume file storage.
  • Formula fields consume 0 storage.
When NOT to Use Certain Fields
  • Avoid multi-select picklist for reporting heavy org.
Avoid formula if value needed in filter frequently (index issue).
  • Avoid lookup if strict dependency needed ‚Üí use master-detail.
