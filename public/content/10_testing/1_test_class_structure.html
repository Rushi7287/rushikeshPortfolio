30-Second Interview Answer

A Test Class in Apex is used to validate business logic and ensure code quality before deployment. It

must be annotated with @isTest, should not use real org data (SeeAllData=false), must create its own
  • test data, and achieve at least 75% code coverage for deployment to Production.
2-Minute Technical Explanation
  • A proper Test Class structure includes:
  • @isTest annotation
  • Separate test methods
  • Test data creation
Test.startTest() and Test.stopTest()
  • Assertions
  • Negative & positive scenarios
  • Basic structure:
  • @isTest
public class MyClassTest {
  • @isTest
static void testMethodName() {
  • // 1. Prepare test data
Account acc = new Account(Name = 'Test Acc');
insert acc;
  • // 2. Execute logic
Test.startTest();
MyClass.myMethod(acc.Id);
Test.stopTest();
  • // 3. Verify result
Account result = [SELECT Name FROM Account WHERE Id = :acc.Id];
System.assertEquals('Expected Value', result.Name);
}
}
Real-Time Scenario
  • You wrote a trigger that updates Opportunity stage.
  • Test class must:
  • Insert Account
  • Insert Opportunity
  • Update record
  • Verify trigger logic executed
  • Without proper assertions ‚Üí poor test quality.

üî∑ 10.b Test.startTest() / Test.stopTest()
üé§INTERVIEW SECTION
30-Second Interview Answer
Test.startTest() and Test.stopTest() are used in Apex test classes to reset governor limits and force
asynchronous code (like Future, Queueable, Batch, Scheduled Apex) to execute synchronously during
  • testing. They ensure accurate testing of real execution behavior.
2-Minute Technical Explanation
  • When writing test classes:
  • Setup code consumes governor limits.
  • Async code does not execute automatically.
  • Limits are shared in the same transaction.
Test.startTest():
  • Resets governor limits.
  • Marks the start of execution block.
Test.stopTest():
  • Executes async jobs.
  • Ends execution block.
Example:
Test.startTest();
MyClass.myMethod();
Test.stopTest();
Without stopTest(), async jobs won‚Äôt run.
Real-Time Scenario
  • You enqueue a Queueable from trigger.
  • If test class doesn‚Äôt use stopTest:
  • Queueable won‚Äôt execute.
  • Coverage incomplete.
  • Assertions fail.
  • Proper testing requires stopTest.
Tricky Interview Questions
  • 1Does startTest reset static variables?
No.
  • 2Does startTest create new transaction?
No, just resets limits.
  • 3What executes at stopTest?
‚Üí All async jobs queued after startTest.
  • 4Can we use multiple startTest blocks?
Only one pair per test method.
  • 5Does stopTest wait for Batch completion?
‚Üí Yes.

üî¨DEEP TECHNICAL BREAKDOWN
Why startTest / stopTest Needed
Problem:
  • Test data setup uses SOQL & DML.
  • Then business logic runs.
  • Limits may already be partially consumed.
Solution:
  • Reset limits before actual logic.
Governor Limit Behavior
  • Before startTest:
  • Limits consumed normally.
  • After startTest:
  • Limits reset to fresh set.
Example:
System.debug(Limits.getQueries()); // 5
Test.startTest();
System.debug(Limits.getQueries()); // 0
Async Execution Behavior
  • Async methods:
  • Future
  • Queueable
  • Batch
  • Scheduled
  • Do NOT run automatically in test.
  • They run at stopTest.
Future Method Example
Test.startTest();
MyFutureClass.myFutureMethod();
Test.stopTest();
  • Future method runs during stopTest.
Queueable Example
Test.startTest();
System.enqueueJob(new MyQueueable());
Test.stopTest();
  • Queueable executes during stopTest.
Batch Example
Test.startTest();
Database.executeBatch(new MyBatch(), 200);
Test.stopTest();
  • Entire batch runs before stopTest completes.
Scheduled Apex Example
String cron = '0 0 2 * * ?';
Test.startTest();
System.schedule('TestJob', cron, new MyScheduler());
Test.stopTest();
  • Scheduled job executes at stopTest.
Important Rules
‚úî Only one startTest/stopTest pair per method
‚úî Must call stopTest to execute async
‚úî Limits reset only once
‚úî Static variables NOT reset
Common Pattern
  • // Setup Data
Test.startTest();
  • // Execute actual logic
Test.stopTest();
  • // Assertions
  • Assertions come AFTER stopTest to verify async results.
What Happens If Not Used?
  • Async code not executed
  • Coverage incomplete
  • Tests pass falsely
  • Integration logic not validated
Static Variable Behavior
Important:
  • Static recursion flags do NOT reset after startTest.
  • Be careful in recursion testing.
Advanced Insight
  • All async jobs enqueued between startTest and stopTest:
  • Execute in order
  • Each runs in separate simulated transaction
  • But nested async from async may not execute further.
Enterprise Best Practices
  • Keep startTest close to execution logic
  • Don‚Äôt put setup inside startTest
  • Always assert results after stopTest
  • Test failure scenarios too
Common Interview Scenario
  • ‚ÄúWhy is my Queueable not running in test class?‚Äù
Answer:
Because stopTest was not called, and asynchronous jobs only execute during Test.stopTest().
