üî∑ 7.f Database.Stateful

üé§INTERVIEW SECTION
30-Second Interview Answer

Database.Stateful is used in Batch Apex to preserve instance variable values across multiple batch

executions. By default, each batch execution is stateless, but implementing Database.Stateful allows

maintaining state, such as counters or aggregated values, throughout the entire batch job.

2-Minute Technical Explanation
  • In Batch Apex:
Each execute() runs in a separate transaction.
  • Instance variables reset after each batch.
  • To retain values across batches, implement Database.Stateful.
Example:
  • public class MyBatch implements
  • Database.Batchable,
Database.Stateful {
Integer totalProcessed = 0;
public void execute(Database.BatchableContext bc, List<Account> scope){
totalProcessed += scope.size();
}
public void finish(Database.BatchableContext bc){
System.debug('Total processed: ' + totalProcessed);
}
}
  • Without Database.Stateful:
  • totalProcessed would reset every batch.
Real-Time Scenario
Requirement:
  • Process 1 million records and:
  • Count total failures
  • Count total successes
  • Send summary email at end
  • Without Stateful:
  • You cannot maintain counters across chunks.
  • With Stateful:
You track totals and send final report in finish().

(Future vs Queueable vs Batch vs Scheduled)
üé§INTERVIEW SECTION
30-Second Interview Answer

Use Future for simple async tasks like basic callouts, Queueable for complex async processing with

chaining and structured logic, Batch Apex for processing large data volumes in chunks, and Scheduled

Apex for running jobs at specific times. In modern development, Queueable is preferred over Future.

2-Minute Technical Explanation
  • Choosing the correct async mechanism depends on:
  • Data volume
  • Complexity
  • Need for chaining
  • Need for scheduling
  • Governor limit concerns
Use Future When:
  • Simple async callout
  • Lightweight processing
  • No chaining required
  • Small data volume
Use Queueable When:
  • Complex logic
  • Need chaining
  • Passing sObjects
  • Integration handling
  • Modern best practice
Use Batch When:
  • Processing > 50,000 records
  • Nightly bulk jobs
  • Large data cleanup
  • Need chunk-based processing
Use Scheduled When:
  • Recurring job
  • Nightly execution
  • Periodic integration
  • Auto-triggering batch
Real-Time Scenarios
  • Scenario 1:
  • Opportunity Closed ‚Üí Send to ERP
Use Queueable
  • Scenario 2:
  • Update 3 million Accounts nightly
Use Scheduled + Batch
  • Scenario 3:
  • Simple callout from trigger
‚ö† Future (or Queueable preferred)
  • Scenario 4:
  • Every day at 2 AM recalculate commissions

8.a Database Methods (Partial Success)
üé§INTERVIEW SECTION
30-Second Interview Answer
Database methods like Database.insert(), Database.update(), and Database.delete() allow partial success

when the allOrNone parameter is set to false. Unlike standard DML, which rolls back the entire

transaction if one record fails, Database methods can process successful records while capturing errors

  • for failed ones using Database.SaveResult.
2-Minute Technical Explanation
  • Normal DML:
insert accList;
  • If 1 record fails ‚Üí
Entire transaction rolls back.
  • Database method:
Database.insert(accList, false);
  • If 1 record fails ‚Üí
Failed records return error details.
Returns:
  • Database.SaveResult[]
  • Each SaveResult contains:
isSuccess()
getErrors()
getId()
  • This is critical in:
  • Bulk data processing
  • Integration handling
  • Batch Apex
  • Error logging systems
Real-Time Scenario
Requirement:
  • Insert 500 Accounts from external system.
  • Some records missing required field.
  • Business requirement:
  • Valid records must save.
  • Invalid records must log error.
Solution:
Use Database.insert(list, false)
  • Log failures in custom object.
Tricky Interview Questions
  • 1What is default behavior of DML?
‚Üí allOrNone = true.
  • 2Does Database.insert(false) prevent trigger execution?
‚Üí No.
  • 3Does partial success avoid rollback?
‚Üí Yes (only for failed records).
  • 4Can partial success be used in trigger?
‚Üí No (not recommended; trigger rollback still applies).
  • 5Does SaveResult give record ID?
‚Üí Yes (for successful records).
Senior-Level Discussion Points
  • Error aggregation pattern
  • Retry mechanism
  • Integration fault tolerance
  • Partial DML in Batch Apex
