30-Second Interview Answer

Chaining Jobs in Salesforce refers to executing one asynchronous job after another, typically using

Queueable Apex or Batch Apex. It is used when processing must occur in sequence, such as multi-step

integrations or staged data processing. Queueable supports direct chaining, while Batch can chain jobs

in the finish() method.
2-Minute Technical Explanation
  • Chaining is required when:
  • Step 1 must complete before Step 2
  • API call depends on previous response
  • Large job broken into logical stages
  • Data transformation followed by integration
  • Two main types of chaining:
Queueable Chaining
Inside execute():
System.enqueueJob(new NextQueueable());
  • Only ONE job can be chained per execution.
Batch Chaining
Inside finish():
Database.executeBatch(new NextBatch());
  • Used for sequential batch processes.

Why Chaining Is Needed
  • Single async job may:
  • Exceed CPU
  • Require dependent API calls
  • Need staged execution
  • Require sequential processing
  • Chaining splits logic cleanly.
Queueable Chaining Example
  • Step 1: First Job
public class FirstQueueable implements Queueable {
public void execute(QueueableContext context){
  • // Step 1 logic
System.enqueueJob(new SecondQueueable());
}
}
  • Step 2: Second Job
public class SecondQueueable implements Queueable {
public void execute(QueueableContext context){
  • // Step 2 logic
}
}
  • Each job runs in separate transaction.
Important Queueable Chaining Rule
  • Inside one execute:
  • Only ONE enqueue allowed.
  • If you try:
System.enqueueJob(new A());
System.enqueueJob(new B());
‚Üí LimitException.
Batch Chaining Example
public class FirstBatch implements Database.Batchable<sObject> {
public void finish(Database.BatchableContext bc){
Database.executeBatch(new SecondBatch());
}
}
  • Second batch starts after first fully completes.
Governor Limits Behavior
  • Each chained job:
  • New transaction
  • Fresh governor limits
  • Separate CPU
  • Separate heap
  • Huge benefit for heavy processing.
Failure Handling Strategy
  • If first job fails:
  • Second job never enqueued
  • Entire chain stops
  • Best practice:
  • Wrap logic in try-catch
  • Log errors
  • Enqueue error handler job if needed
AsyncApexJob Monitoring
  • Each job creates record in:
  • AsyncApexJob object.
Track:
  • Status
  • JobType
  • ParentJobId
Bad ‚ùå
  • Job A ‚Üí enqueues Job B
  • Job B ‚Üí enqueues Job A
  • Infinite loop.
  • Always design controlled chain.
Chaining Depth
  • Salesforce allows deep chaining but must avoid:
  • 100+ chained jobs
  • Hitting daily async limits
  • Long dependency chains
  • Design carefully.
Real Enterprise Pattern
  • Complex ERP Sync:
  • Queueable 1 ‚Üí Validate Data
  • Queueable 2 ‚Üí Send API
  • Queueable 3 ‚Üí Process Response
  • Queueable 4 ‚Üí Update Records
  • Queueable 5 ‚Üí Notify Admin
  • Clean modular architecture.
When NOT to Use Chaining
  • Avoid chaining if:
  • Independent tasks
  • No dependency
  • Can process in one batch
  • Simpler logic sufficient
Queueable vs Batch Chaining
  • Feature Queueable Batch
  • Best For Sequential steps Large volume stages
Where to chain execute() finish()
  • Data Volume Medium Very large
  • Control High High
