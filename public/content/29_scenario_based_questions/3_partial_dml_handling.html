29.c Partial DML Handling

üé§INTERVIEW SECTION
30-Second Interview Answer

Partial DML handling in Salesforce allows some records in a bulk operation to succeed even if others fail,

using Database methods like Database.insert(records, false) to avoid full transaction rollback.
2-Minute Technical Explanation
  • Default DML behavior:
insert recordList;
  • If one record fails:
Entire transaction rolls back.
  • To allow partial success:
Use:
Database.insert(recordList, false);
  • Second parameter:
allOrNone = false
Now:
‚úî Valid records are inserted
‚úî Invalid records return error
‚úî Transaction does not fully rollback
  • Critical in:
‚úî Bulk integrations
‚úî Data migration
‚úî Batch Apex
‚úî API-based record creation
Real-Time Scenario
  • External system sends:
  • 200 Account records.
  • 5 records fail validation rule.
  • If using normal insert:
All 200 fail.
  • If using partial DML:
‚úî 195 succeed
‚úî 5 fail
‚úî Errors logged
  • Integration becomes resilient.

üî¨DEEP TECHNICAL BREAKDOWN
Default DML Behavior
insert accounts;
  • If one fails ‚Üí
  • Entire transaction rolled back.
  • This is:
  • Atomic behavior.
  • Good for:
  • Business integrity.
  • Bad for:
  • Bulk integrations.
Using Partial DML
Syntax:
  • Database.SaveResult[] results =
Database.insert(accounts, false);
false = allow partial success.
Handling SaveResult
for(Integer i = 0; i < results.size(); i++){
if(results[i].isSuccess()){
System.debug('Inserted: ' + results[i].getId());
}
else{
for(Database.Error err : results[i].getErrors()){
System.debug('Error: ' + err.getMessage());
}
}
}
Important:
  • Always handle errors.
Database Methods Supporting Partial DML
‚úî Database.insert()
‚úî Database.update()
‚úî Database.upsert()
‚úî Database.delete()
  • All support allOrNone parameter.
Partial DML in Batch Apex
  • Common pattern:
In execute():
Database.update(scope, false);
Allows:
‚úî Some records succeed
‚úî Others logged
‚úî Batch continues
  • Without this:
  • Batch fails entirely.
When NOT to Use Partial DML
  • If transaction must be atomic:
Example:
  • Create Order + Payment + Invoice.
  • If any fails:
  • All must rollback.
  • Then use default DML.
Integration Architecture Pattern
  • External System ‚Üí
  • Queueable/Batch ‚Üí
  • Partial DML ‚Üí
  • Log errors in custom object ‚Üí
  • Retry failed records later.
  • Enterprise pattern.
Partial DML vs try-catch
  • try-catch:
  • Catches entire DML failure.
  • Partial DML:
  • Handles per-record failure.
  • Different use cases.
Interview Scenario
  • ‚ÄúHow would you handle inserting 500 records where some may fail validation?‚Äù
Answer:
I would use Database.insert(recordList, false) to allow partial success, then iterate over the SaveResult

array to identify failed records, log the errors, and optionally implement a retry mechanism.
