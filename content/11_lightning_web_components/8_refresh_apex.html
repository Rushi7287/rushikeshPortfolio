30-Second Interview Answer
refreshApex() is used in LWC to refresh cached data returned by a @wire service after performing a data

update. It forces the wired Apex method or LDS adapter to re-execute and fetch the latest data from the

  • server.
2-Minute Technical Explanation
When using @wire, Salesforce caches data (if cacheable=true).
After:
  • Record update
  • Record delete
  • Imperative Apex DML
  • The wired data does NOT automatically refresh.
  • To refresh:
  • Ûæ†Æ Store wired result reference
Ûæ†Ø Call refreshApex(wiredResult)
Example:
import { refreshApex } from '@salesforce/apex';
@wire(getAccounts)
wiredAccounts(result) {
this.wiredResult = result;
if(result.data){
this.accounts = result.data;
}
}
  • After update:
refreshApex(this.wiredResult);
  • Now latest data is fetched.
Real-Time Scenario
  • Component displays Account list using @wire.
User clicks Delete button (Imperative call).
  • Without refreshApex:
  • Deleted record still visible.
  • With refreshApex:
  • List reloads and UI updates.

Why refreshApex Needed?
Because:
  • @wire uses client-side caching.
  • After DML via Imperative call:
  • Cache still contains old data.
  • UI shows stale information.
  • refreshApex forces fresh query.
Correct Pattern
  • Step 1: Wire Data
@wire(getAccounts)
wiredAccounts(result) {
this.wiredResult = result;
if(result.data){
this.accounts = result.data;
}
}
  • Step 2: Perform Imperative DML
deleteAccount({ accountId: this.selectedId })
.then(() => {
return refreshApex(this.wiredResult);
});
  • Refresh happens after delete completes.
Important Rule
  • You must pass:
refreshApex(this.wiredResult);
NOT:
refreshApex(this.accounts); ‚ùå
  • Because refresh requires wired result object.
refreshApex with LDS
  • If using:
@wire(getRecord, {...})
record;
  • You can refresh same way:
refreshApex(this.record);
Common Bug
  • Developer writes:
refreshApex(getAccounts); ‚ùå
  • Wrong.
  • Must store wire result reference.
Performance Consideration
Avoid:
  • Calling refresh after every small change
  • Refresh inside renderedCallback
  • Multiple refresh calls unnecessarily
  • Best practice:
  • Refresh only after DML success.
Advanced Pattern ‚Äì Optimistic UI
  • Instead of refresh:
  • Remove record from UI immediately
  • Then call refreshApex to sync
  • Improves UX.
üìå refreshApex vs location.reload()
  • Never use:

  • Bad UX.
  • refreshApex is lightweight and efficient.
Enterprise Pattern
  • Dashboard component:
  • Wire loads data
  • User updates record
  • Imperative save
  • refreshApex reloads
  • Toast message shows success
  • Clean user flow.
Limitations
  • Works only with wired data
  • Not for imperative-only variables
  • Requires proper reference storage

30-Second Interview Answer
Lightning Message Service (LMS) enables communication between different Lightning components
(LWC, Aura, Visualforce) across the DOM without a parent-child relationship. It uses a message channel
  • to publish and subscribe to messages in a loosely coupled way.
2-Minute Technical Explanation
  • Normally in LWC:
  • Parent ‚Üí Child via @api
  • Child ‚Üí Parent via events
  • But what if:
  • Components are not related?
  • On different parts of page?
  • LWC needs to talk to Aura or Visualforce?
  • Solution ‚Üí LMS.
Steps:
  • Ûæ†Æ Create Message Channel
  • Ûæ†Ø Import in component
  • Ûæ†∞ Publish message
  • Ûæ†± Subscribe to message
Uses:
  • lightning/messageService
  • Best for:
‚úî Cross-component communication
‚úî Utility bar communication
‚úî Console apps
‚úî Independent UI blocks
Real-Time Scenario
  • You have:
Account List Component (left panel)
Account Detail Component (right panel)
  • When user clicks account in list ‚Üí
  • Detail component updates automatically.
  • No parent-child relation.
  • Use LMS.
