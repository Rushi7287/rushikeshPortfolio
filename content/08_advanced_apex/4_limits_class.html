üî∑ 8.d Limits Class

üé§INTERVIEW SECTION
30-Second Interview Answer

The Limits class in Apex provides methods to monitor governor limit consumption in real time, such as

SOQL queries, DML statements, CPU time, and heap size. It helps developers write optimized, bulk-safe

  • code and prevent limit exceptions in complex transactions.
2-Minute Technical Explanation
  • Salesforce enforces governor limits per transaction.
  • The Limits class allows you to:
  • Check how many SOQL queries used
  • Check remaining CPU time
  • Check heap usage
  • Monitor DML usage
  • Prevent hitting limits proactively
Example:
System.debug(Limits.getQueries());
System.debug(Limits.getLimitQueries());
  • Useful in:
  • Complex trigger frameworks
  • Large batch processing
  • Integration-heavy logic
  • Performance optimization
Real-Time Scenario
  • You have:
  • Trigger
  • Flow
  • Workflow
  • Integration
  • Transaction CPU approaching 10 sec.
  • You use:
if(Limits.getCpuTime() > 8000){
System.enqueueJob(new AsyncJob());
}
  • Move heavy logic to async before hitting limit.
Tricky Interview Questions
  • 1Does Limits class increase limits?
No.
  • 2Can we reset limits using Limits?
No.
  • 3Is limit same in sync and async?
‚Üí No (async higher CPU, SOQL).
  • 4How to check remaining SOQL?
‚Üí Limits.getLimitQueries() - Limits.getQueries()
  • 5Does rollback reset limits?
Senior-Level Discussion Points
  • Proactive limit management
  • Async deferral strategy
  • Dynamic bulk control
  • Logging near-limit scenarios
  • LDV optimization
  • Heap size monitoring
üî¨DEEP TECHNICAL BREAKDOWN
Why Limits Class Exists
  • Salesforce multi-tenant architecture:
  • Shared resources
  • Strict per-transaction limits
  • Limits class provides visibility.
SOQL Queries
Limits.getQueries();
Limits.getLimitQueries();
  • Sync limit: 100
  • Async limit: 200
DML Statements
Limits.getDmlStatements();
Limits.getLimitDmlStatements();
Limit: 150
CPU Time
Limits.getCpuTime();
Limits.getLimitCpuTime();
Sync: 10,000 ms
Async: 60,000 ms
Heap Size
Limits.getHeapSize();
Limits.getLimitHeapSize();
Sync: 6 MB
Async: 12 MB
Callouts
Limits.getCallouts();
Limits.getLimitCallouts();
Limit: 100
Example ‚Äì Defensive Programming
if(Limits.getQueries() > 90){
System.debug('Approaching SOQL limit');
}
  • Used in complex frameworks.
CPU Guard Pattern
if(Limits.getCpuTime() > 9000){
System.enqueueJob(new HeavyProcessingQueueable());
}
  • Prevents runtime limit exception.
Limit-Aware Batch Example
if(Limits.getHeapSize() > 5000000){
break;
}
  • Stop processing to avoid heap exception.
Governor Limit Differences
  • Limit Sync Async
  • SOQL 100 200
  • CPU 10 sec 60 sec
  • Heap 6MB 12MB
  • Limits class reflects correct context automatically.
Important Clarifications
  • Limits class is read-only.
  • Cannot increase limits.
  • Cannot reset limits.
  • Works per transaction.
Enterprise Use Case
  • Complex Trigger Framework:
  • Multiple handler classes
  • Multiple object updates
  • Integration logic
  • Add monitoring layer:
  • Log when usage exceeds threshold.
  • Helps production debugging.
Debugging Near-Limit Issues
  • If CPU error:
  • 1Add debug using Limits.getCpuTime()
  • 2Identify heavy loops
  • 3Move logic to Queueable
  • 4Reduce nested loops
When NOT to Overuse Limits Class
Avoid:
  • Over-checking inside every loop
  • Micro-optimizing unnecessarily
  • Writing defensive code without real need
  • Use strategically.
Interview Power Line
  • If interviewer asks:
  • ‚ÄúHow do you prevent hitting governor limits in complex logic?‚Äù
Answer:

By bulkifying code properly and using the Limits class to monitor SOQL, DML, CPU time, and heap usage

  • so that heavy processing can be deferred to asynchronous Apex before limits are exceeded.

üî∑ 8.e Schema Methods (Dynamic Apex)
üé§INTERVIEW SECTION
30-Second Interview Answer

Schema methods in Dynamic Apex allow us to access metadata information at runtime, such as object

names, field names, record types, picklist values, and field-level security. They are useful when building

reusable, configurable, or metadata-driven solutions without hardcoding object or field references.

2-Minute Technical Explanation
  • Dynamic Apex enables:
  • Getting object metadata at runtime
  • Getting field details dynamically
  • Checking CRUD & FLS
  • Fetching Record Type IDs without hardcoding
  • Reading picklist values dynamically
  • Core namespace:
  • Schema
  • Common usage:
  • Schema.SObjectType.Account
  • Schema.SObjectType.Account.fields.Name
  • Used in:
  • Generic frameworks
  • Integration engines
  • Metadata-driven apps
  • Admin-configurable logic
  • Reusable utilities
Real-Time Scenario
Requirement:
  • Build generic logging framework that works for ANY object.
  • Instead of hardcoding:
  • acc.Name
Use:
sObj.get('Name');
  • And use Schema methods to validate fields dynamically.
Tricky Interview Questions
  • 1How do you get RecordTypeId dynamically?
‚Üí Using getRecordTypeInfosByName().
  • 2How to get picklist values dynamically?
‚Üí getDescribe().getPicklistValues().
  • 3Can Schema methods bypass FLS?
‚Üí No.
  • 4What is difference between DescribeSObjectResult and DescribeFieldResult?
‚Üí Object-level vs field-level metadata.
  • 5Does Dynamic Apex affect performance?
‚Üí Slight overhead, use carefully.
Senior-Level Discussion Points
  • Metadata-driven architecture
  • Dynamic SOQL + Schema combo
  • Caching describe results
  • Generic trigger framework
  • Field-level security enforcement
  • Admin-configurable logic

What is Dynamic Apex?
  • Dynamic Apex allows runtime:
  • Object discovery
  • Field discovery
  • Metadata inspection
  • Security checking
Opposite of static Apex (hardcoded fields).
Describe Object Example
  • Schema.DescribeSObjectResult accDescribe =
Account.SObjectType.getDescribe();
System.debug(accDescribe.getLabel());
Provides:
  • Label
  • API name
  • Key prefix
  • Createable / Updateable
  • Accessible
Describe Field Example
  • Schema.DescribeFieldResult fieldDesc =
Account.Name.getDescribe();
System.debug(fieldDesc.getType());
System.debug(fieldDesc.isAccessible());
System.debug(fieldDesc.isCreateable());
  • Used for CRUD/FLS enforcement.
Getting All Fields of Object
  • Map fields =
Schema.SObjectType.Account.fields.getMap();
Loop:
for(String fieldName : fields.keySet()){
System.debug(fieldName);
}
  • Useful for dynamic frameworks.
Dynamic Record Type Retrieval
  • Avoid hardcoding:
Bad:
acc.RecordTypeId = '012XXXXXXXXXXXX';
Good:
  • Map rtMap =
Schema.SObjectType.Account.getDescribe()
.getRecordTypeInfosByName();

  • Safe across environments.
Dynamic Picklist Values

Account.Industry.getDescribe();

fieldDesc.getPicklistValues();
for(Schema.PicklistEntry val : pickVals){

}
  • Used in LWC controllers.

.get('Account')
.newSObject();
acc.put('Name', 'Dynamic Account');
insert acc;
  • Very powerful in integration frameworks.

if(Schema.sObjectType.Account.isCreateable()){
insert acc;
}

  • Combined with Schema for dynamic field selection.
Enterprise Use Case

  • Admin selects:
  • Object

  • Use dynamic only when necessary.
