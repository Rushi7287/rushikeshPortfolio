30-Second Interview Answer

Access modifiers in Apex control the visibility and accessibility of classes, methods, and variables. The

main access modifiers are public, private, protected, and global. Public allows access within the org,

private restricts access within the class, protected allows access in subclasses, and global allows access

  • across namespaces, especially for managed packages.
2-Minute Technical Explanation
  • Access modifiers define who can see and use your code.
üîπ private (Default)
  • Accessible only within the same class.
  • Most restrictive.
  • Default if no modifier is specified.
public
  • Accessible anywhere within the same org.
  • Most commonly used.
protected
  • Accessible within defining class and its subclasses.
  • Mainly useful in inheritance.
global
  • Accessible across namespaces.
  • Required for managed packages.
  • Most permissive.
Real-Time Scenario
  • You are building a utility class for internal use only:
Use:
  • private methods for helper logic.
  • If building reusable service class used by multiple LWC controllers:
Use:
  • public methods.
  • If creating managed package:
Use:
  • global.

private
  • Default modifier.
class MyClass {
private void helperMethod() {
System.debug('Internal');
}
}
  • Accessible only inside MyClass.
public
  • Accessible across entire org.
public class MyService {
public static void process() {
System.debug('Accessible anywhere');
}
}
  • Can be called from:
  • Triggers
  • LWC controller
  • Anonymous window
  • Other classes
protected
  • Used in inheritance.
public virtual class ParentClass {
protected void display() {
System.debug('Parent method');
}
}
public class ChildClass extends ParentClass {
public void callMethod() {
display();
}
}
  • Accessible only in subclass.
global
  • Used in managed packages.
global class MyPackageClass {
global static void execute() {
System.debug('Accessible outside namespace');
}
}
  • Required when:
  • Exposing Apex REST from managed package
  • Making class accessible to subscriber orgs

Performance & Security Consideration
  • Bad Practice:
  • Using global everywhere.
  • Exposing unnecessary methods.
  • Good Practice:
  • Restrict visibility.
  • Follow principle of least privilege.
  • Use private helper methods.

Debugging Visibility Issues
  • If method not accessible:
  • 1Check class modifier.

When NOT to Use global
  • Do not use global unless:
  • Building managed package
  • Exposing public API
  • Required for integration
  • Using global unnecessarily increases security exposure.
