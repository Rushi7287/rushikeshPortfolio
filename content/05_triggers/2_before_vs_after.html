Before Vs After

5.b Before vs After

üé§INTERVIEW SECTION
30-Second Interview Answer

Before triggers are used to modify field values before records are saved to the database, while after

triggers are used when you need the record ID or want to perform actions like creating related records.

Before triggers do not require additional DML to update the same record, making them more efficient.

2-Minute Technical Explanation
  • Difference depends on execution timing in order of execution.
Before Trigger
  • Runs before record is committed to database.
  • Used for:
  • Setting default values
  • Validation logic
  • Updating same record fields
Preventing save (addError)
Example:
trigger AccountBefore on Account (before insert) {
for(Account acc : Trigger.new){
acc.Description = 'Created via Trigger';
}
}
  • No update statement required.
After Trigger
  • Runs after record is saved.
  • Used for:
  • Creating related records
  • Updating related objects
  • When record ID required
  • Integration call preparation
Example:
trigger AccountAfter on Account (after insert) {
List<Contact> contacts = new List<Contact>();
for(Account acc : Trigger.new){
contacts.add(new Contact(
LastName = acc.Name,
AccountId = acc.Id
));
}
insert contacts;
}
  • ID available only after insert.

Performance Comparison
Before:
  • No extra DML
  • Lower CPU usage
  • More efficient
After:
  • Requires DML for same record update
  • Higher CPU consumption
  • Risk of recursion

Recursion Risk Example
  • After Update:
  • Update child record
  • Child trigger updates parent
  • Parent trigger updates child
‚Üí Infinite loop.
  • Use static variable control.
Record Locking Consideration
  • After trigger:
  • Record already locked.
  • Updating related parent may cause lock contention.
  • Before trigger:
  • Safer for same record modifications.
Governor Limit Awareness
Both:
  • Share same transaction limits.
  • SOQL limit = 100.
  • DML limit = 150.
  • After triggers more likely to hit DML limit due to extra operations.
Bulk Insert
  • Both before and after receive full list.
Partial Success

If one record fails ‚Üí whole transaction fails unless Database methods used outside trigger.

Flow Interaction
  • Flow may run after before trigger and before after trigger depending on type.
Debugging Before vs After Issues
  • If field not updating:
  • 1Check event type.
  • 2Check if DML required.
  • 3Check recursion.
  • 4Check order of execution.
  • 5Check Flow overriding value.
When NOT to Use After
  • Avoid after if:
  • Only modifying same record.
  • No need for ID.
  • No related object creation.
  • Use before instead.
When NOT to Use Before
  • Avoid before if:
  • Creating related records.
  • Need record ID.
Performing callout (must be async after commit).
