üé§INTERVIEW SECTION
30-Second Interview Answer

Recursion in Salesforce triggers occurs when a trigger causes an update that fires the same trigger

again, potentially leading to infinite loops and governor limit exceptions. Recursion is commonly

controlled using static variables or proper logic checks to ensure the code runs only once per

  • transaction.
2-Minute Technical Explanation
  • Recursion happens when:
  • A trigger updates the same object.
  • A child trigger updates parent.
  • Flow + Trigger interaction.
  • Process Builder updates same record.
  • Integration updates record again.
Example:
  • After Update Trigger:
trigger AccountTrigger on Account (after update) {
update Trigger.new; // Dangerous
}
  • This causes infinite recursion.
Solution:
  • Use static boolean flag in handler class.
Real-Time Scenario
Requirement:
  • When Account Industry changes:
‚Üí Update related Contacts.
  • But updating Contacts triggers another Account update via automation.
  • Without recursion control:
  • CPU limit exceeded.
  • With recursion control:
  • Safe execution.
Tricky Interview Questions
  • 1Why does recursion happen?
‚Üí Because update inside trigger fires trigger again.
  • 2Does static variable persist across transactions?
‚Üí ‚ùå No (only per transaction).
  • 3Will recursion happen in before trigger?
‚Üí Yes, if DML executed.
  • 4Does Flow also cause recursion?
‚Üí Yes.
  • 5Can we rely only on static Boolean?
‚Üí Not always (complex scenarios require better pattern).
Senior-Level Discussion Points
  • Static Set pattern
  • Field change detection before update
  • Idempotent logic design
  • Transaction boundary awareness
  • Flow + Trigger order interaction
  • Queueable separation for recursion control

Basic Recursion Control Pattern
Static Boolean Pattern (Simple Case)
public class AccountHandler {
private static Boolean isRunning = false;
public static void handle(){
if(isRunning) return;
isRunning = true;
  • // logic
}
}
  • Works for simple recursion.
Better Pattern ‚Äì Static Set
  • Boolean fails in bulk scenarios sometimes.
  • Better approach:
public class AccountHandler {
private static Set<Id> processedIds = new Set<Id>();
public static void afterUpdate(List<Account> newList){
for(Account acc : newList){
if(!processedIds.contains(acc.Id)){
processedIds.add(acc.Id);
  • // logic
}
}
}
}
  • Handles per-record recursion safely.
Field Change Detection Pattern
  • Best practice:
  • Only run logic when field changed.
for(Account acc : Trigger.new){
Account oldAcc = Trigger.oldMap.get(acc.Id);
if(acc.Industry != oldAcc.Industry){
  • // execute logic
}
}
  • Prevents unnecessary update.
üìå Idempotent Logic (Advanced Concept)
  • Design logic so:
  • Even if executed twice ‚Üí no harmful effect.
Example:
  • Instead of blindly updating field:
if(acc.Status__c != 'Processed'){
acc.Status__c = 'Processed';
}
  • Safe even if trigger fires again.
Async Recursion Handling
  • If integration must update record:
  • Instead of direct update:
System.enqueueJob(new MyQueueable(accIds));
  • Queueable runs in new transaction.
  • Avoids same transaction recursion.
Why Static Works?
Because:
  • Static variable shared across transaction.
  • Reset after transaction ends.
  • Prevents multiple execution in same transaction.
Limitations of Static Boolean
  • Fails when:
  • Multiple logical operations needed.
  • Multiple events handled.
  • Multiple objects involved.
Better:
  • Use static Set.
Recursion in Batch Apex
  • Each batch execution:
  • New transaction.
  • Static resets.
  • Must handle carefully.
Debugging Recursion Issues
  • If CPU limit exceeded:
  • 1Check debug log for repeated trigger execution.
  • 2Search for update statements.
  • 3Check Flow interaction.
  • 4Check Workflow field updates.
  • 5Add debug markers to track execution count.
Recursion vs Order of Execution
Important:
  • Workflow field update:
  • Triggers run again.
  • Flow after save:
  • Triggers may fire again.
  • Must understand full execution lifecycle.
When NOT to Use Static Boolean Alone
  • Avoid only Boolean when:
  • Handling multiple update scenarios.
  • Bulk record updates.
  • Complex parent-child logic.
  • Use combination:
  • Field change detection
  • Static Set
  • Async separation

üé§INTERVIEW SECTION
30-Second Interview Answer

Order of Execution in Salesforce defines the exact sequence of steps that occur when a record is saved.

It includes validation rules, before triggers, after triggers, workflow rules, processes, flows, and

database commit. Understanding it is critical to avoid recursion, unexpected field overwrites, and

  • integration failures.
2-Minute Technical Explanation

When a record is saved, Salesforce executes multiple layers of automation in a predefined order.

  • High-level flow:
  • 1System Validation
  • 2Before Triggers
  • 3Custom Validation Rules
  • 4After Triggers
  • 5Assignment Rules
  • 6Auto-response Rules
  • 7Workflow Rules
  • 8Process Builder / Flow (After-save)
  • 9Roll-up Summary Calculations
  • 10Commit to Database
  • 11Post-commit logic (Async jobs, Emails, Future, Queueable)
  • Understanding this is crucial when:
  • Field value gets overwritten
  • Trigger fires twice
  • Recursion happens
  • Flow interacts with trigger
  • Mixed automation causes conflict
Real-Time Scenario
  • You update Opportunity Stage to Closed Won.
  • You have:
  • Before trigger updating Amount
  • After trigger creating Project
  • Workflow updating CloseDate
  • Flow sending email
  • If CloseDate changes via workflow:
  • Trigger runs again.
  • Without understanding order ‚Üí unexpected recursion.
Tricky Interview Questions
  • 1Does workflow field update cause trigger to fire again?
‚Üí Yes.
  • 2Does commit happen before or after workflow?
‚Üí After workflow.
  • 3When does ID become available?
‚Üí After insert, before after trigger.
  • 4When does roll-up summary recalculate?
‚Üí After save.
  • 5Do async methods run before commit?
‚Üí No (after commit).
Senior-Level Discussion Points
  • Flow before-save vs after-save difference
  • Post-commit logic
  • Record locking timing
  • Trigger re-entry due to workflow
  • Performance optimization strategy
  • Designing automation hierarchy

Why Triggers Fire Twice
  • If workflow updates a field:
Example:
  • Account saved ‚Üí Workflow sets Rating ‚Üí
  • That update triggers before update and after update again.
  • Developers must:
  • Detect field change.
  • Use recursion control.
Before-Save Flow vs After-Save Flow
  • Before-save flow:
  • Runs before before trigger?
  • No ‚Üí runs after system validation but before before trigger completes.
  • After-save flow:
  • Runs after after trigger but before commit.
  • Important for interviews.
Record Locking
  • During save:
  • Record locked.
  • Related parent may also lock.
  • Data skew can cause contention.
Governor Limit Awareness
  • Entire order runs in one transaction.
  • Same CPU time.
  • Same SOQL count.
  • Same DML count.
  • Unless async.
Rollback
  • If exception thrown in after trigger:
  • Entire transaction rolled back.
Partial DML Outside Trigger
  • Only possible using Database methods in Apex.
Mixed Automation
  • Trigger + Flow + Workflow:
  • Complex recursion risk.
Debugging Order Issues
  • If field overwritten:
  • 1Check validation rules.
  • 2Check workflow field updates.
  • 3Check Flow.
  • 4Check after trigger.
  • 5Check debug execution order.
Best Practices
  • Prefer before trigger for same-record updates.
  • Avoid workflow field updates if trigger handles logic.
  • Avoid mixing Flow + Trigger for same logic.
  • Use async for heavy processing.
  • Use clear automation ownership strategy.
