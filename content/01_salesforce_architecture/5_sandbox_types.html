30-Second Interview Answer

Salesforce provides different sandbox types to support development and testing: Developer, Developer

Pro, Partial Copy, and Full Sandbox. They differ in storage capacity, data availability, and refresh

interval. Developer sandboxes are mainly for coding, Partial for testing with sample data, and Full

  • sandbox is a complete copy of Production used for UAT and performance testing.
2-Minute Technical Explanation

A Sandbox is a copy of your Production org used for development and testing without affecting live

  • users.
  • Salesforce offers four main types:
  • Ûæ†Æ Developer Sandbox
  • Copy of metadata only
  • No production data
  • 200 MB data storage
Refresh: Once per day
  • Used for: Development, unit testing
  • Ûæ†Ø Developer Pro Sandbox
Metadata + more data storage (1 GB)
  • No production data copy
Refresh: Once per day
  • Used for: Development with larger test data
  • Ûæ†∞ Partial Copy Sandbox
Metadata + selected production data (via template)
  • 5 GB data storage
Refresh: Every 5 days
  • Used for: QA, integration testing
  • Ûæ†± Full Sandbox
Exact replica of Production (metadata + all data)
  • Same storage as Production

  • Used for: UAT, performance testing, staging

What Gets Copied?
  • Always Copied:
  • Custom Objects
  • Fields
  • Apex Classes
  • Triggers
  • Flows
  • Validation Rules
  • Only in Partial/Full:
  • Selected or full data
  • Reports
  • Dashboards with data
Internal Working of Refresh
  • When you click Refresh:
  • 1Salesforce locks sandbox.
  • 2Copies metadata from production.
  • 3Copies selected data (if applicable).
  • 4Generates new org ID.
  • 5Users reset passwords.
  • Old sandbox data erased.
Data Security Consideration
  • Full sandbox contains:
  • Real customer data
  • PII
  • Financial data
  • So you must:
  • Mask sensitive data
  • Disable outbound integrations
  • Change endpoints
  • Restrict user access
Performance Considerations
  • Full sandbox useful for:
  • Large data volume testing
  • Query performance testing
  • Batch performance validation
  • Load testing
  • Developer sandbox cannot simulate production performance.
Integration Still Pointing to Production
  • If endpoints not changed ‚Üí sandbox can send data to live system.
User Emails Sent from Sandbox
  • Can accidentally send emails to real customers.
Solution:
  • Disable email deliverability
  • Use email relays carefully
Best Practices
  • Use Developer sandbox per developer.
  • Use Partial for QA.
  • Use Full for UAT.
  • Never test performance in Dev sandbox.
  • Refresh sandboxes regularly.
Use sandbox naming convention (DEV1, QA1, UAT1).
When NOT to Use Full Sandbox
  • Small UI change
  • Minor validation update
  • Early development
Cost-sensitive environments (Full is expensive)
