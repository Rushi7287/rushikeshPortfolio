30-Second Interview Answer

Test Data Creation is the practice of creating all required records inside a test class instead of relying on

existing org data. Best practice is to use SeeAllData=false, create minimal but complete data, and

  • optionally use a Test Data Factory pattern for reusable and maintainable test setup.
2-Minute Technical Explanation
  • In Apex testing:
  • You should NOT rely on real org data.
  • Default is SeeAllData=false.
  • All necessary objects must be created manually.
  • Create only required fields.
  • Respect validation rules.
  • Respect relationships.
  • Best practice:
‚úî Create required parent records
‚úî Use factory class for reuse
‚úî Bulk-create test data
‚úî Avoid unnecessary queries
  • Poor test data design causes:
  • Test failures in production
  • Deployment issues
  • Validation rule conflicts

Parent-Child Example
  • Opportunity requires:
  • Account
  • StageName
  • CloseDate
Account acc = new Account(Name='Test');
insert acc;
Opportunity opp = new Opportunity(
Name='Test Opp',
StageName='Prospecting',

);

  • Must include required fields.
Using @TestSetup
  • Used for common data across test methods.
  • @isTest
public class MyTest {
  • @TestSetup
static void setupData(){
insert new Account(Name='Common');
}
  • @isTest
static void test1(){
Account acc = [SELECT Id FROM Account LIMIT 1];
}
}
  • Data created once per test class.
  • Improves performance.
Test Data Factory Pattern
  • Reusable utility class.
public class TestDataFactory {
public static Account createAccount(){
Account acc = new Account(Name='Test');
insert acc;
return acc;
}
public static Opportunity createOpportunity(Id accId){
Opportunity opp = new Opportunity(
Name='Test Opp',
StageName='Prospecting',
CloseDate=Date.today(),
AccountId=accId
);
insert opp;
return opp;
}
}
  • Used in multiple test classes.
  • Clean architecture.

  • Tests fail in production because:

  • Validation rule added
  • Field made mandatory
  • Test data must satisfy validations.
