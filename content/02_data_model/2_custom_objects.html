üé§INTERVIEW SECTION
30-Second Interview Answer

Custom Objects are user-defined database tables in Salesforce created to store data that standard

objects cannot handle. They allow full customization, including custom fields, relationships, validation

  • rules, triggers, and automation. Custom objects have API names ending with __c.
2-Minute Technical Explanation
  • When standard objects don‚Äôt satisfy business requirements, we create Custom Objects.
Example:
  • If a company wants to track:
  • Projects
  • Assets
  • Training Programs
  • Site Visits
  • These may not fit into standard CRM objects.
  • So we create:
  • Project__c
  • Training__c
  • Asset__c
  • Custom objects:
  • Are fully customizable
  • Support relationships
  • Support triggers
  • Support record types
  • Can be deployed
Can be deleted (unlike standard objects)
Real-Time Scenario
  • Company wants to track:
  • Construction Sites.
  • You create:
  • Site__c
  • Engineer__c
  • Inspection__c
  • Then create:
  • Lookup relationship between Inspection and Site
  • Trigger to auto-update site status
  • Flow for approval
  • All done using custom objects.

What Is a Custom Object Internally?
  • Custom object is:
  • Metadata definition
  • Table in Salesforce DB
  • Has OrgId segregation
  • Supports standard CRUD operations
  • Internally stored similar to standard objects but defined in org metadata.

Best Practices
  • Use standard objects first.
  • Create custom only when needed.
  • Use meaningful API names.
  • Enable reporting option.
  • Use external IDs for integrations.
  • Avoid unnecessary relationships.
When NOT to Use Custom Object
If:
  • Standard object fits 80% requirement
  • You can extend using custom fields
  • Process aligns with CRM
  • Then prefer standard.
