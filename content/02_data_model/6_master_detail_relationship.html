30-Second Interview Answer

A Master-Detail relationship is a tightly coupled relationship between two objects where the child record

is completely dependent on the parent. The child inherits the parent‚Äôs sharing and security, and deleting

the parent automatically deletes all related child records. Master-Detail also enables Roll-Up Summary

  • fields.
2-Minute Technical Explanation
  • Master-Detail is used when:
  • Child record cannot exist without parent
  • Strong data integrity is required
  • Roll-Up Summary is needed
  • Sharing must be inherited
  • Key characteristics:
  • Parent field is mandatory
Cascade delete enabled (automatic)
  • Child inherits Owner from parent
  • Child inherits OWD from parent
  • Supports Roll-Up Summary
Example:
  • OpportunityLineItem ‚Üí Opportunity
  • Invoice_Line__c ‚Üí Invoice__c
Real-Time Scenario
  • You build:
Invoice__c (Parent)
Invoice_Line__c (Child)
  • Business rule:
  • Invoice line cannot exist without Invoice
  • If Invoice deleted ‚Üí delete all lines
  • Invoice should show total amount of all lines
Solution:
  • Use Master-Detail relationship and Roll-Up Summary.

What Happens Internally
  • Master-Detail field stores:
  • Parent Id
  • Foreign key
  • Enforces referential integrity
  • When parent deleted:
Platform automatically deletes all children (cascade).
Structural Example
Parent:
  • Invoice__c
Child:
  • Invoice_Line__c
Field:
Invoice__c (Master-Detail)
  • Child table includes:
Invoice__c (foreign key, required)
Sharing Behavior
  • Feature Behavior
  • Owner Inherited from parent
  • OWD Inherited
  • Sharing Rules Controlled by parent
  • Manual Sharing Not available
Roll-Up Summary
  • Only available in Master-Detail.
Types:
  • COUNT
  • SUM
  • MIN
  • MAX
Example:
  • Invoice__c.Total_Amount__c
= SUM(Invoice_Line__c.Amount__c)
  • Calculated automatically.
SOQL Relationship Queries
  • Parent to Child
SELECT Name, (SELECT Name FROM Invoice_Lines__r)
  • FROM Invoice__c
  • Child to Parent
  • SELECT Name, Invoice__r.Name
  • FROM Invoice_Line__c
Reparenting Option
  • If enabled:
  • Child can change parent.
  • If disabled:
  • Parent cannot be changed after creation.
Performance Considerations
  • ‚ö† Data Skew Issue:
  • If 100,000 child records under one parent:
  • Lock contention
  • Slow updates
  • Sharing recalculation delay
  • Best practice:
  • Distribute child records across multiple parents if possible.
Deleting Parent
  • Deletes all child records permanently.
Converting Lookup ‚Üí Master-Detail
Conditions:
  • No orphan records
  • Child OWD must be Public Read/Write
Large Data Volume
  • Heavy roll-up recalculation may:
  • Increase CPU time
  • Slow record save
Debugging Master-Detail Issues
  • If record not saving:
  • 1Check parent field mandatory.
  • 2Check parent existence.
  • 3Check sharing inherited.
  • 4Check reparenting setting.
  • 5Check roll-up summary recalculation delay.
Master-Detail vs Lookup Quick Comparison
  • Feature Lookup Master-Detail
  • Parent Required Optional Required
  • Cascade Delete Optional Always
When NOT to Use Master-Detail
  • Do NOT use if:
  • Child needs independent sharing
  • Child may exist without parent
  • Ownership must differ
  • Large data skew risk
  • Then prefer Lookup.
